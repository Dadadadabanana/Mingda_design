Problems

UX team frequently faced design and development requests from across the entire company. However, disorganised requests and arbitrary changes often disrupted team members' work cadence.

Solutions

To address pain points in managing UX requests, I designed a standardised UX request management platform that tracked request progress, balanced team workload, and showcased team value.

Project Goals

How might we design an internal management platform that standardises request management, allocates resources efficiently , and clearly showcases the UX team's value.

Launch Results

The platform launched successfully after a month of testing, with the UX team using it to manage requests and other stakeholders cooperating well. But much of the communication around requests remained offline.

Design Process

I interviewed team members to understand their frustrations, mapped workflows, structured the IA, and designed a solution to standardise request management, continuously improving it.

Request Journey

Building on the existing process, I streamlined and formalised the request journey, designing a clear status flow to guide how each request moves through the system.

Roles and Permissions

One of the most challenging parts of the platform was designing the permission structure. I defined clear user roles based on the request workflow and tailored pages and access levels for each role.

Information Architecture

Our platform's information architecture was structured into three sections: Requests, Teams, and Statistics. Each page included relevant primary actions and approval functions.

Challenges

After the initial launch, the platform's functionality remained basic, we encountered several challenges, particularly with the flow of processes, page usability, and demonstrating team value. I continued to optimise these areas in following iterations.

Key Pages

Create a request

A "Create Request" button was placed in the top navigation bar so users could quickly create a request from anywhere.

The form opened without navigating away from the current page, helping users stay oriented.

After submission, users were directed to the "To Be Confirmed" request list, clearly showing the request status.

Request in progress

Clearly display request status and progress of contributors at each stage.

Request details

The request drawer displayed request information and action buttons based on different user roles and permissions.

Schedule a new item

Each stage had its own scheduling option, allowing users to add or edit their timelines flexibly, with a "TBC" (To Be Confirmed) option available.

Team

Team members' current tasks were clearly displayed and clickable to view all requests they were involved in.

Dashboard

The dashboard provided an intuitive overview of request distribution and resource allocation.

Key Learning

When collaborating with product managers, I proactively arranged cross-team meetings and clearly outlined the support needed while providing all necessary documentation. This aligned stakeholder expectations and clarified responsibilities.

Future Considerations

Some key workflows like request creation, scheduling, and approvals still need clearer structure. A central space for tracking request communications on the platform would improve usability.
